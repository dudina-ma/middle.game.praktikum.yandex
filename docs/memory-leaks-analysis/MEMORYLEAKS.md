# Отчёт о проверке утечек памяти

**Дата:** 09–10 марта 2026 года
**Версия приложения:** коммит 940cc928b0379e239d0b290c31a238f5c8457a2e

## 1. Методология

Тестирование проводилось разными способами, ниже - отчёт по каждому из них.

### 1.1. Memory panel и сравнение снимков

- Chrome DevTools (Memory panel), метод сравнения снимков (comparison snapshots).
- Действия (переходы между страницами, полный цикл игры) повторялись 3–5 раз для исключения случайных флуктуаций GC.
- Для ключевых сценариев также использовалась запись Allocation timeline.

По результатам сравнения снимков **не обнаружено** типичных признаков утечек:

- **Detached DOM trees** — поиск по «Detached» не выявил узлов, отцепленных от документа, но удерживаемых в памяти.
- **Собственные компоненты** — конструкторы React-компонентов (Profile, Leaderboard и т.д.) после возврата на главную не накапливались, количество обнулялось.
- **Обработчики событий и таймеры** — не выявлено удержания контекстов (system/Context) или крупных массивов, которые должны были быть освобождены после ухода со страницы.

### 1.2. Вкладка Performance

Использовалась вкладка **Performance** с включённым чекбоксом **Memory** (запись при навигации и действиях пользователя). Резких скачков памяти не выявлено.

### 1.3. Анализ кода на утечки памяти

Проведён статический анализ кода (подписки, таймеры, слушатели событий, очистка при размонтировании).

#### Игровой модуль (Game, GameController, InputManager, сцены)

| Компонент | Подписки / таймеры | Очистка | Статус |
|-----------|--------------------|---------|--------|
| **InputManager** | `mousedown`, `mousemove`, `contextmenu` на canvas | `destroy()` вызывает `removeEventListener` для всех трёх и `callback = null` | ✅ |
| **GameController** | `store.on('update', this.update)` | `destroy()` вызывает `store.off('update', this.update)`, `inputManager.destroy()`, `curentScene.destroy()`, `finishGame = null` | ✅ |
| **Game** | `store.on('update', this.update)`, `requestAnimationFrame` в цикле `render()` | `destroy()` вызывает `cancelAnimationFrame(this.frame)`, `store.off('update', this.update)`, `gameController.destroy()` | ✅ |
| **BattleScene** | `store.on('update', this.update)`, `setTimeout` для AI хода | `destroy()` вызывает `clearTimeout(this.timeout)`, `store.off('update', this.update)`, выставляет `isDestroyed = true` | ✅ |
| **SetupScene** | `store.on('update', this.update)` | `destroy()` вызывает `store.off('update', this.update)` | ✅ |

Цепочка вызовов при уходе со страницы игры или смене фазы: **GameBoard** (cleanup `useEffect`) → `gameInstance.current.destroy()` → **Game.destroy()** → **GameController.destroy()** → **InputManager.destroy()** и **curentScene.destroy()**. Все слушатели и таймеры снимаются.

#### React-компоненты и хуки

| Место | Подписки | Очистка | Статус |
|-------|----------|---------|--------|
| **GameBoard** | Создаёт `Game`, запускает `render()` (requestAnimationFrame) | В cleanup `useEffect`: `gameInstance.current.destroy()`, `gameInstance.current = null` | ✅ |
| **OfflineBanner** | `window.addEventListener('online'/'offline')` | cleanup `useEffect`: `removeEventListener` для обоих | ✅ |
| **useFullscreenState** | `document.addEventListener` для fullscreen-событий (4 варианта префиксов) | cleanup `useEffect`: `removeEventListener` для всех | ✅ |

#### Найденные проблемы (исправлено)

- **main.tsx:** Обработчик `window.addEventListener('load', …)` для регистрации Service Worker не снимался после срабатывания — оставалась лишняя подписка на всё время жизни вкладки. **Исправлено:** в начале колбэка вызывается `removeEventListener('load', onLoad)`, обработчик снимается после первого вызова.

## 2. Memlab

### 2.1. Контекст

При попытке протестировать веб-приложение с помощью Memlab возникла проблема: инструмент открывает новый экземпляр браузера в режиме инкогнито с чистым профилем, где отсутствуют все cookies и данные сессии. Приложение требует авторизации через 3rd-party cookies от внешнего домена (.ya-praktikum.tech), а фронтенд работает на localhost.

### 2.2. Попытки решения

**Прямая установка cookies через API Memlab**

- Пытались передать готовые cookies из существующей сессии.
- **Не сработало:** браузер автоматически менял домен cookie на localhost из-за политики same-origin.

**Использование Chrome DevTools Protocol (CDP)**

- Пытались установить cookies напрямую через CDP в обход ограничений.
- **Не сработало:** CDP сессия наследует контекст текущей страницы, поэтому cookies всё равно привязывались к localhost.

**Создание iframe с доменом API**

- Пытались создать iframe с доменом .ya-praktikum.tech, чтобы установить cookies в правильном контексте.
- **Сработало частично:** cookies действительно установились на нужный домен, но сервер вернул ошибку 401 с сообщением "Cookie is not valid".

**Диагностика API ответов**

- Проанализировали ответы сервера на запросы авторизации.
- Выяснили, что проблема не в способе установки cookie, а в её невалидности (сессия истекла или была привязана к другим параметрам).

**Полноценный логин в рамках теста**

- Пытались выполнить логин в сценарии Memlab (ввод логина/пароля, отправка формы), чтобы получить свежую сессию.
- **Не сработало:** в режиме инкогнито кука после успешного ответа сервера не устанавливается (или не сохраняется в контексте), поэтому авторизованная сессия не появляется.

## 3. Вывод

По результатам проверки утечек памяти инструментами Chrome DevTools (Memory panel, сравнение снимков, Performance с Memory) и статического анализа кода приложение в целом ведёт себя стабильно: типичные признаки утечек (detached DOM-деревья, накопление экземпляров компонентов, удержание обработчиков и таймеров) не выявлены, резких скачков памяти при навигации не зафиксировано. Игровой модуль и React-компоненты корректно снимают подписки и очищают ресурсы при размонтировании; единственная обнаруженная проблема с обработчиком `load` в main.tsx исправлена. Тестирование с Memlab для авторизованных сценариев не удалось провести из-за ограничений работы cookies в инкогнито и с внешним API.
